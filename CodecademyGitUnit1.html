Basic Git Workflow

Git is the industry standard version control system for web developers.

A project managed with Git is called a Git repository.

GitHub is a popular hosting service for Git repositories. As GitHub is in the cloud, you can back-up your files, share your code, and collaborate with others.

git init initializes a new project, essentially creating the working directory.
git add filename puts filename into the staging area.
git add filename1 filename2 puts two files into staging.
git status shows what is in the staging area.
git commit -m "comment" moves everything in the staging area to the local repository.

git diff filename checks the differences between the working directory and the staging area.
git log shows the history of commits. The log is in chronological order with the most recent commit at the top. Each commit entry contains:

  • A 40 character hash code (called SHA) that uniquely identifies the commit
  • The commit author
  • The date and time of the commit
  • The commit message

BackTracking

Backtracking differs from making a new commit. By backtracking instead of making new commits, we’re able to make small changes to our previous commits without cluttering our repository history with commit messages similar to “forgot to add file X from my previous commit so doing that now.” It’s important to understand that when you backtrack to amend your last commit, you’re not so much fixing it as replacing it entirely with a new, improved commit that pushes the old commit out of the way and puts the new commit in its place. Effectively, it’s as if the previous commit never happened, and it won’t show up in your repository history.

In Git, the commit you are currently on is known as the HEAD commit.

To see the HEAD commit, enter git show HEAD The output will display everything the git log command displays for the HEAD commit, plus all of the file changes that were commited.

Say that you've made a change to a file in your working directory, but decide that you want to go back to how you had it before the last commit (having forgotten the exact differences). Typing git checkout HEAD filename will change your working copy back. (shortcut: git checkout -- filename )

Say that you've added a file to staging, continued to make changes to the file in the working directory, and then realize that you shouldn't have added that file to staging. Typing git reset HEAD filename will reset the file in staging to be the same as the HEAD commit. It does not discard file changes from the working directory, it just removes them from the staging area.

Say that you've commited several times and realize you would like to restart at a point several commits back. Type git reset commit_SHA where commit_SHA is the first seven characters of the 40 character hash code that corresponds to the last commit that you would like to return to. HEAD is now set to that previous commit.
